# Промпт для глубокого исследования: UI/визуальный редизайн игры «Gen»

> **Как использовать.** Этот документ — задание (brief) для глубокого многоисточникового
> исследования. Цель исследования — дать обоснованную, со ссылками, базу знаний и набор
> конкретных рекомендаций, которые позволят спроектировать UI игры «Gen» так, чтобы он
> максимально соответствовал своему жанру, и при этом был готов к интеграции графических
> ассетов, генерируемых строго ограниченным числом «спрайт-карт».
>
> Результат — отчёт (Markdown) со структурой из раздела «9. Ожидаемые результаты».

---

## 1. Контекст: что за игра

**Gen** — браузерная (2D, single-player, `localStorage`) **тактическая RPG с пошаговыми боями
на квадратной сетке** (4 ортогональных направления, манхэттен-дистанция, line-of-sight для
дальних атак) и **глубокой мета-прогрессией «Memento Mori»** (прогресс не обнуляется при
поражении; сила копится по осям «смерть / использование / победа»).

Жанровые ориентиры для понимания типа: Fire Emblem, Final Fantasy Tactics, Into the Breach,
Wildermyth, плюс мета-прогрессия в духе Slay the Spire / roguelite-метагейма.

Ключевые экраны и сущности (из спецификации):

- **Campaign Hub** с вкладками: Персонаж (roster до 100, отряд из 4 слотов, экипировка из 3
  слотов, умения, пассивы, сундук), Магазин, Таверна (найм), Кодекс (энциклопедия ~168
  записей в 7 категориях), Справка, и вкладка Бой/Экспедиция (выбор из 7 цепочек).
- **Battle Screen**: поле-сетка переменного размера (от 1×2 до 20×20), очередь хода по
  инициативе, бейджи порядка хода на клетках, панель действий активного юнита (умения с
  кулдаунами, базовая атака, перемещение, конец хода), журнал боя, экраны победы/поражения.
- **Inter-Battle Screen**: между боями экспедиции (revive downed-бойцов и т.п.).
- **Контент-таксономия, влияющая на ассеты:** 8 классов героев (warrior, mage, ranger,
  healer, rogue, paladin, warlock, berserker); 8 рас врагов (beast, undead, human, orc, elf,
  specter, construct, demon); 8 боссов (по 1 на класс); 9 базовых статов (каждый сейчас
  обозначен эмодзи); 11 видов умений (`CardKind`: melee, ranged, aoe, heal, regen, resurrect,
  buff, debuff, dot, lifesteal_spell, utility); 3 слота экипировки; 15 склонностей;
  система модификаторов (моды) со слотами; статус-эффекты (баффы/дебаффы).
- **Memento Mori** — центральная фишка: тема смерти/памяти/накопления силы. Это сильный
  тематический якорь для арт-направления.

**Текущее состояние UI (что улучшаем):** функциональный, но «дефолтный» Ant Design 6,
тёмная тема, **эмодзи вместо иконок** юнитов/статов, поле боя — простые CSS-`div`-клетки
44px с эмодзи и HP-полосой, никакого выраженного визуального стиля и атмосферы. Нужно
поднять до уровня, узнаваемо принадлежащего жанру тактической RPG с мрачной мета-темой.

Полный контекст по механикам и сущностям — в файлах репозитория `gen_spec_for_claude.md`
(спецификация) и `implementation-report.md` (что реализовано/заглушки). Исследователь должен
опираться на них при оценке информационной плотности UI.

## 2. Жёсткие ограничения проекта (важнее «красоты»)

Эти ограничения **первичны**. Любая рекомендация по стилю/компоновке, которая их нарушает,
бесполезна. Исследование должно рассматривать всё через их призму.

1. **Технологический стек фиксирован:** React 19 + **Ant Design 6** + Zustand 5 + Vite 6.
   Решение по UI-фреймворку принято: **AntD остаётся, перекрашиваем темой** (ConfigProvider
   theme tokens + кастомный CSS/слой стилей поверх). Никакого ухода на другой UI-kit.
   → Исследование должно учитывать **возможности и пределы системы темизации AntD 6**
   (design tokens, algorithm dark, component tokens, CSS-in-JS, `cssVar`) и как через неё
   достичь нужной атмосферы, не переписывая компоненты.

2. **🔴 ГЛАВНОЕ ОГРАНИЧЕНИЕ — генерация ассетов «картами спрайтов»:** графика генерируется
   AI как **целые спрайт-атласы на одном изображении** (sprite sheet / atlas / contact
   sheet), а **не поэлементно**. Бюджет — **максимум 3–5 запросов генерации карт-атласов**
   на всю игру. Нет возможности итеративно догенерировать отдельные иконки/портреты.
   → Это диктует выбор арт-стиля (он должен быть **когерентным и воспроизводимым в рамках
   одного промпта на целый лист из десятков элементов**), таксономию ассетов (как упаковать
   все нужные элементы в 3–5 листов), и стиль, прощающий неидеальную консистентность между
   листами.

3. **Формат ассетов:** растровые PNG (AI image generation). Нужны рекомендации по
   разрешению листов, размеру ячейки, прозрачности/фону, нарезке (slicing) атласа в
   спрайты, и по тому, как сетка элементов на листе должна быть устроена, чтобы её можно
   было детерминированно нарезать кодом (CSS sprite / background-position, либо нарезка в
   отдельные файлы на этапе сборки).

4. **Охват:** все экраны, **desktop-first** (адаптив/мобайл — вторичны, но не ломать).

5. **Русскоязычный UI**, высокая информационная плотность (много статов, тултипов,
   бейджей) — стиль не должен жертвовать читабельностью ради атмосферы.

## 3. Главные вопросы исследования

Исследование должно дать обоснованные ответы на следующее.

### 3.1. Жанровый UX тактических RPG
- Какие устоявшиеся UI/UX-паттерны характерны для пошаговых тактических RPG на сетке
  (отображение поля, подсветка перемещения/дальности/LoS, очередь хода/таймлайн
  инициативы, индикация активного юнита, превью урона/попадания, панель действий,
  лог боя)? Что считается «обязательным», а что — лишним?
- Как лучшие представители жанра решают **высокую информационную плотность** (статы,
  статус-эффекты, кулдауны, бейджи очереди) не перегружая экран?
- Паттерны UI для **мета-прогрессии и roguelite-хабов** (экраны прокачки, выбор
  модификаторов/карт, инвентарь, древо/слоты улучшений) — на что опереться для Memento
  Mori, слотов модов (бейдж «M+»), кодекса.

### 3.2. Арт-направление, СОВМЕСТИМОЕ с генерацией атласами
Это центральный, наиболее ценный блок. Нужно **сравнить кандидаты арт-стилей именно по
критерию «насколько хорошо стиль генерируется и держит консистентность в режиме одного
большого спрайт-листа за 1 промпт»**, а не только по эстетике.

Кандидаты для сравнения (минимум): тёмное/готическое фэнтези, пиксель-арт (16/32-bit),
рисованное painterly-фэнтези (карточный арт в духе Slay the Spire), плоский
вектороподобный/иконографический стиль, силуэтный/монохром-с-акцентом. Для каждого оценить:

- **Воспроизводимость в атласе:** держит ли стиль единый вид при генерации десятков
  элементов на одном листе одним промптом? Насколько прощает мелкие расхождения?
- **Консистентность между 3–5 разными листами** (портреты, иконки, фоны, токены) —
  насколько легко удержать единый стиль в нескольких отдельных генерациях.
- **Тематическое соответствие** мрачной теме Memento Mori и фэнтези-бестиарию (8 рас,
  боссы, нежить/демоны).
- **Читабельность в мелком размере** (токены юнитов на клетке ~44px, иконки статов,
  иконки умений) — критично для тактической сетки.
- **Техническая дружелюбность:** чистый фон/прозрачность, чёткие границы для нарезки,
  отсутствие «перетекания» элементов друг в друга на листе.
- **Риски AI-генерации:** где стиль обычно «плывёт» (лица, текст, мелкие иконки, единый
  масштаб) и как стиль помогает/мешает это скрыть.

Итог блока — **рекомендованный арт-стиль + 1–2 запасных**, с явной аргументацией от
ограничения «3–5 атласов».

### 3.3. Таксономия и упаковка ассетов в 3–5 атласов
- Какие визуальные ассеты вообще нужны этой игре (составить полный инвентарь:
  токены/портреты 8 классов и 8 рас + 8 боссов; иконки 9 статов; иконки 11 видов умений
  и/или конкретных умений; иконки 3 слотов экипировки и предметов; иконки склонностей и
  модов; статус-эффекты; элементы поля боя — тайлы пола/стен/зон спавна/подсветки;
  фоны экранов хаба; UI-рамки/панели/кнопки/бейджи; логотип/заставка)?
- **Как сгруппировать этот инвентарь в 3–5 связных листов** так, чтобы (а) элементы
  одного листа естественно генерировались одним промптом в едином стиле и масштабе,
  (б) число листов уложилось в бюджет. Предложить конкретную раскладку, например:
  «Лист 1 — портреты персонажей/врагов (сетка N×M), Лист 2 — иконки статов+умений+статусов,
  Лист 3 — тайлы поля и подсветки, Лист 4 — UI-рамки/кнопки/панели (9-slice), Лист 5 —
  фоны экранов». Обосновать группировку и оценить, что важнее, если урезать до 3.
- **Спецификация под нарезку:** рекомендации по размеру листа, сетке (число рядов/колонок,
  паддинги между ячейками), размеру ячейки под каждый тип, прозрачному фону, и по тому,
  что включать прямо в промпт генерации, чтобы лист был ровной сеткой, пригодной для
  автоматической нарезки.
- Какие элементы UI лучше **оставить за CSS/AntD-темой** (градиенты, тени, рамки,
  типографика), а не тратить на них бюджет генерации.

### 3.4. Темизация Ant Design 6 под выбранный стиль
- Как через design tokens AntD 6 (seed/map/alias tokens, `theme.darkAlgorithm`,
  component-level tokens, `cssVar`) задать палитру, типографику, радиусы, тени, плотность,
  соответствующие выбранному арт-направлению.
- Где AntD-компоненты придётся «обходить» кастомным слоем (например, поле боя, токены
  юнитов, карточки персонажей с портретами, рамки в стиле ассетов) и как это сделать,
  не ломая остальной AntD-UI.
- Подбор шрифтов (с поддержкой кириллицы!) под жанр и стиль; иконочная стратегия
  (когда свои спрайт-иконки статов/умений, когда @ant-design/icons).
- Учесть известные нюансы AntD 6 из отчёта (deprecations: `Divider`, `List`, `Space`/`Flex`
  `direction`) — чтобы рекомендации не опирались на устаревшее API.

### 3.5. Маккапы и места под ассеты (asset-ready layout)
- Как спроектировать раскладку **каждого экрана** так, чтобы заранее заложить
  «посадочные места» (slots/placeholders) под будущие ассеты с фиксированными
  соотношениями сторон и размерами, чтобы вставка нарезанных спрайтов не ломала вёрстку.
- Рекомендованный механизм плейсхолдеров на время до генерации (заглушки с подписями
  размеров/имён ассетов, либо текущие эмодзи как временный слой) и как организовать код,
  чтобы переключение «эмодзи → спрайт-атлас» было точечным (один компонент-обёртка,
  маппинг id→позиция в атласе).
- Конкретные ASCII/схематичные маккапы ключевых экранов с разметкой зон под ассеты:
  как минимум **Battle Screen**, **Hub / вкладка Персонаж**, **Таверна (карточка
  кандидата)**, **экран выбора мода / слотов Memento**. Для каждой зоны указать:
  тип ассета, из какого атласа, целевой размер/aspect ratio.

### 3.6. Читабельность, доступность, перфоманс
- Контраст и читабельность мелких токенов/иконок на тёмном фоне; цветовая
  дифференциация свой/враг/downed/подсветка с учётом дальтонизма (не только цветом).
- Перфоманс: спрайт-атласы vs отдельные файлы, lazy-load фонов, влияние на текущий
  бандл (отчёт упоминает 892 КБ одним чанком) — рекомендации, чтобы графика не
  усугубила.

## 4. Что обязательно изучить (источники)

- Разборы UI/UX тактических пошаговых RPG (Fire Emblem, FF Tactics, Into the Breach,
  Tactical Breakthrough, Wildermyth, Battle Brothers) — статьи, GDC-доклады, UX-разборы.
- UI roguelite/deckbuilder с мета-прогрессией (Slay the Spire, Hades, Monster Train) —
  как подаются выбор апгрейдов, слоты, инвентарь.
- Практики **создания и нарезки спрайт-атласов** (texture atlas / sprite sheet): размеры,
  паддинг, power-of-two, CSS sprites, инструменты нарезки.
- Практики **генерации игровых ассетов через AI картами/контакт-листами**: промптинг для
  единого стиля на одном листе, удержание консистентности между листами, типичные сбои,
  работа с прозрачным фоном, ограничения по тексту/лицам/мелким иконкам.
- Официальная **документация Ant Design 6**: система токенов, кастомизация темы,
  dark algorithm, component tokens, миграция/deprecations 6.x. (Использовать актуальную
  документацию, а не память — версии меняются.)
- Доступность интерфейсов с цветовым кодированием (WCAG контраст, дальтонизм).

## 5. Глубина и метод

- Многоисточниковое исследование с перекрёстной проверкой ключевых утверждений.
- Для арт-стилей и режима «генерация атласом» — особенно критично проверять заявления
  на реальных примерах/гайдах, т.к. это главное ограничение проекта.
- Где есть выбор (стиль, группировка атласов, шрифты) — давать **рекомендацию с
  обоснованием**, а не только перечень опций. Указывать trade-off.
- Все нетривиальные утверждения — со ссылками на источники.

## 6. Явно НЕ в scope

- Смена UI-фреймворка (AntD остаётся — решено).
- Игровая логика, баланс, механики боя (это в спецификации, не предмет UI-исследования).
- Мультиплеер, серверная часть, монетизация, мобильная нативная сборка.
- Поэлементная (по одному ассету) генерация графики — противоречит ограничению.
- Аудио, анимации боя как отдельная большая тема (можно упомянуть места под них, но не
  исследовать глубоко).

- Ассеты — **AI-генерация растровых изображений целыми атласами** (3–5 запросов).

9. **Пошаговый план внедрения** — порядок: тема AntD → плейсхолдеры/слоты → промпты
   генерации атласов → нарезка → интеграция, по экранам.
10. **Открытые вопросы и риски** + список источников.
